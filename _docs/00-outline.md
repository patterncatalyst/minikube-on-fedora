## Suggested structure

A typical technical tutorial follows this shape:

1. **Prerequisites** — what readers need installed before starting
2. **Introduction / Why this exists** — conceptual grounding before
   any commands
3. **Hello world** — the simplest possible working example
4. **Core concepts** — one section per major idea, building up
5. **Realistic example** — putting the concepts together
6. **Debugging** — what to do when things go wrong
7. **Production considerations** — what changes for real workloads
8. **Where to go next** — pointers to deeper resources

Adjust the count up or down based on your topic's complexity. The
project this skeleton was extracted from had 17 sections; a simpler
topic might have 6.

## File naming

Tutorial sections live in `_docs/` and are named with a numeric
prefix:

- `00-outline.md` (this file)
- `01-prerequisites.md`
- `02-introduction.md`
- ...

The number controls reading order and becomes part of the URL slug
(e.g., `/docs/01-prerequisites/`). Renaming files lets you reorder
sections, but check internal cross-links if you do.
